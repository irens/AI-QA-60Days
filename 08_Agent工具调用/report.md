# 质量报告 · 08 Agent 工具调用正确性

> 本报告基于**真实模型运行**（deepseek-chat），非模拟数据。

---

## 一、测试目标

给模型工具清单，对 5 个请求输出工具调用决策，核对工具名 + 关键参数。门禁：正确率 ≥ 0.8。

---

## 二、测试结果：✅ 通过（正确率 100%，5/5）

| 请求 | 期望 | 实际决策 | 结果 |
|---|---|---|:--:|
| 北京今天天气怎么样？ | get_weather(北京) | `{"tool":"get_weather","args":{"city":"北京"}}` | ✅ |
| 查一下上海的天气 | get_weather(上海) | `{"tool":"get_weather","args":{"city":"上海"}}` | ✅ |
| 算 23 乘以 47 | calculate(23,47) | `{"tool":"calculate","args":{"expression":"23*47"}}` | ✅ |
| 100 加 200 | calculate(100,200) | `{"tool":"calculate","args":{"expression":"100+200"}}` | ✅ |
| 你好呀，今天心情不错 | none | `{"tool":"none","args":{}}` | ✅ |

三个环节全对：**选对工具、抽对参数、闲聊不乱调**。

**门禁状态**：🟢 放行。

---

## 三、守门人洞察

1. **"不该调时别调"同样重要**：很多 Agent 测试只测"有需求时调对没"，却忽略"闲聊时会不会乱调"。本例专门放了一条闲聊用例守这个边——模型正确判了 `none`。
2. **把自然语言→结构化调用变成可核对的契约**：工具名 + 参数都是结构化的，可逐字段断言，这让"Agent 决策对不对"从玄学变成可量化门禁。
3. **100% 是"happy path"的结论**：真实风险在边界——下一步要测的才是硬骨头。

---

## 四、局限与下一步

- 本批是清晰的"标准请求"。真正的脆弱点在：
  - **参数 Schema 异常**：缺参、类型错、范围越界时，Agent 能否优雅处理而非崩溃；
  - **工具故障注入**：工具超时/报错时，是否有重试与降级；
  - **多步编排**：需要连续调用多个工具时，中间结果传递与错误传播。
- 当前用"提示词输出 JSON"模拟工具调用；可进一步接入原生 function-calling 接口测试。
