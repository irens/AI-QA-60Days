## 🎭 为什么会有这个项目？

**背景：**
我是一名 43 岁、拥有十几年经验的传统自动化测试工程师。面对 AI 浪潮，我不打算去卷底层的算法开发（那不是我的优势）。我选择从 **"系统稳定性、质量风控、缺陷注入、自动化评估"** 的 QA 视角，重新解构 AI 应用。

这个仓库记录了我**用 60 - 80 天时间，每天 1 小时**，从零跑通 AI 系统并建立测试体系的全过程。如果你和我一样有转型焦虑，欢迎 Fork 和 Star，我们一起破局。

### 🗡️ 第一阶段：破除黑盒 —— LLM 工作原理与缺陷注入 (Day 1-12)

*聚焦于搞懂 LLM 的底层机制，并从测试视角找出最容易崩溃的环节。*

> 💡 **QA 洞察**：测试 LLM 不要当成测传统接口！你必须理解**概率性输出**与**确定性测试**的本质差异。

| 进度 | Day | 核心研究主题 | 实验产出与 QA 洞察 |
|:---:|:---:|:---|:---|
| ✅ | **Day 01** | **[温度参数验证](./Phase1_LLM_Foundation/Day01)** | 💡 **QA洞察**：温度=0并非真正确定性！极低温度下的"贪婪解码"可能与预期行为有微妙差异。 |
| ✅ | **Day 02** | **[Tokenizer 与上下文边界](./Phase1_LLM_Foundation/Day02)** | 💡 **QA洞察**：不同Tokenizer的编码差异直接影响成本估算和上下文规划。BPE与SentencePiece对中文处理的压缩效率差异可达30-50%。 |
| ✅ | **Day 03** | **[Top-p 采样与输出多样性](./Phase1_LLM_Foundation/Day03)** | 💡 **QA洞察**：创意任务与事实查询任务的最优温度参数显著不同，必须分类测试。 |
| ✅ | **Day 04** | **[模型版本回归测试](./Phase1_LLM_Foundation/Day04)** | 💡 **QA洞察**：模型版本管理需追踪API版本、模型快照、训练数据截止时间等多个维度。 |
| ✅ | **Day 05** | **[温度×Top-p 参数网格扫描](./Phase1_LLM_Foundation/Day05)** | 💡 **QA洞察**：参数组合效应是输出稳定性的隐形杀手，单一参数测试无法发现组合风险。 |
| ✅ | **Day 06** | **[输出稳定性时间维度验证](./Phase1_LLM_Foundation/Day06)** | 💡 **QA洞察**：同一参数配置在不同时间点的输出一致性，可检测模型版本更新或基础设施变化的影响。 |
| ✅ | **Day 07** | **[模型版本迭代与回归测试](./Phase1_LLM_Foundation/Day07)** | 💡 **QA洞察**：模型版本管理需追踪API版本、模型快照、训练数据截止时间等多个维度。 |
| ✅ | **Day 08** | **[A/B 测试实验设计](./Phase1_LLM_Foundation/Day08)** | 💡 **QA洞察**：样本量计算、显著性水平选择、功效分析是A/B测试的统计基础。 |
| ✅ | **Day 09** | **[幻觉检测与事实一致性](./Phase1_LLM_Foundation/Day09)** | 💡 **QA洞察**：幻觉分为事实性、忠实性、内在三类，需采用不同的检测策略。 |
| ✅ | **Day 10** | **[NLI-based 忠实度评分](./Phase1_LLM_Foundation/Day10)** | 💡 **QA洞察**：基于NLI的entailment检测是验证忠实性的有效手段，但对隐含推理检测不足。 |
| ✅ | **Day 11** | **[输出鲁棒性测试——同义改写](./Phase1_LLM_Foundation/Day11)** | 💡 **QA洞察**：词汇/句法/语篇层面的变换测试，可发现模型对表述方式的过度敏感。 |
| ✅ | **Day 12** | **[输出鲁棒性测试——跨语言与语篇级改写](./Phase1_LLM_Foundation/Day12)** | 💡 **QA洞察**：跨语言场景下翻译往返可能丢失关键信息，语篇级改写可能破坏长距离依赖。 |

### 🛡️ 第二阶段：Prompt 攻防 —— 人机交互入口的安全边界 (Day 13-21)

*聚焦于红蓝对抗（Red Teaming）与大模型的越狱漏洞。*

> 💡 **QA 洞察**：Prompt 是 AI 系统的"需求文档"，必须可测试、可版本管理、可 A/B 对比。

| 进度 | Day | 核心研究主题 | 实验产出与 QA 洞察 |
|:---:|:---:|:---|:---|
| ✅ | **Day 13** | **[直接注入攻击测试](./Phase2_Prompt_Testing/Day13)** | 💡 **QA洞察**：分隔符逃逸、角色覆盖、指令重定向是最常见的三种直接注入模式。 |
| ✅ | **Day 14** | **[间接注入攻击测试](./Phase2_Prompt_Testing/Day14)** | 💡 **QA洞察**：外部数据源（网页、文档）是间接注入的主要载体，需建立内容净化机制。 |
| ✅ | **Day 15** | **[Prompt 结构设计与可测试性原则](./Phase2_Prompt_Testing/Day15)** | 💡 **QA洞察**：将Prompt视为"软代码"，应用软件工程的可测试性原则确保输出稳定性。 |
| ✅ | **Day 16** | **[Prompt 结构可测试性设计](./Phase2_Prompt_Testing/Day16)** | 💡 **QA洞察**：CO-STAR框架（Context-Objective-Style-Tone-Audience-Response）是设计可测试Prompt的有效工具。 |
| ✅ | **Day 17** | **[Few-shot 示例稳定性](./Phase2_Prompt_Testing/Day17)** | 💡 **QA洞察**：示例扰动测试（替换、删除、重排）可验证Prompt对示例选择的敏感度。 |
| ✅ | **Day 18** | **[跨模型 Few-shot 迁移测试](./Phase2_Prompt_Testing/Day18)** | 💡 **QA洞察**：同一套Few-shot示例在不同模型上表现可能差异巨大，多模型部署需验证示例兼容性。 |
| ✅ | **Day 19** | **[系统 Prompt 安全边界测试](./Phase2_Prompt_Testing/Day19)** | 💡 **QA洞察**：系统Prompt是AI应用的"宪法"，一旦被突破，所有安全策略都将失效。 |
| ✅ | **Day 20** | **[注入风险扫描](./Phase2_Prompt_Testing/Day20)** | 💡 **QA洞察**：Prompt注入是AI应用的"SQL注入"，需建立输入过滤、意图分类、输出监控多层防护。 |
| ✅ | **Day 21** | **[Prompt 质量门禁集成](./Phase2_Prompt_Testing/Day21)** | 💡 **QA洞察**：CI/CD流水线中的Prompt质量门禁需覆盖功能、安全、性能三个维度。 |

### 📐 第三阶段：量化玄学 —— RAG 系统的自动化评估标准体系 (Day 22-50)

*解决"如何用数学公式衡量大模型回答质量"的业界难题。*

> 💡 **QA 洞察**：测试 RAG 不要当成测传统接口！你必须把**"检索准不准（Embedding）"**和**"答得对不对（Chat）"**这两个组件解耦来测。

#### 3.1 检索层质量保障 (Day 22-33)

| 进度 | Day | 核心研究主题 | 实验产出与 QA 洞察 |
|:---:|:---:|:---|:---|
| 🔲 | **Day 22** | **[文档质量评分与预处理流程测试](./Phase3_RAG_Reliability/Day22)** | 💡 **QA洞察**：预处理流水线是RAG系统的"安检门"，坏文档入库会导致检索质量灾难性下降。 |
| 🔲 | **Day 23** | **[文档质量门禁与坏文档识别](./Phase3_RAG_Reliability/Day23)** | 💡 **QA洞察**：质量门禁需覆盖格式合规性、内容完整性、语义可读性三个维度，阈值设定需业务校准。 |
| 🔲 | **Day 24** | **[分块策略评估：固定长度 vs 语义分块](./Phase3_RAG_Reliability/Day24)** | 💡 **QA洞察**：固定长度分块实现简单但可能切断语义，语义分块效果更好但计算成本更高，需权衡选择。 |
| 🔲 | **Day 25** | **[递归分块与 Agentic 分块对比](./Phase3_RAG_Reliability/Day25)** | 💡 **QA洞察**：递归分块保持层次结构，Agentic分块利用LLM智能决策，适合对质量要求极高的场景。 |
| 🔲 | **Day 26** | **[Embedding 模型选型：MTEB 评测与领域适配](./Phase3_RAG_Reliability/Day26)** | 💡 **QA洞察**：通用Embedding模型在垂直领域表现可能下降30%+，领域微调是保障检索质量的关键。 |
| 🔲 | **Day 27** | **[向量相似度校准与阈值调优](./Phase3_RAG_Reliability/Day27)** | 💡 **QA洞察**：余弦相似度0.7在不同模型、不同领域含义差异巨大，必须建立业务相关的阈值标准。 |
| 🔲 | **Day 28** | **[向量数据库选型测试：召回率、延迟、并发](./Phase3_RAG_Reliability/Day28)** | 💡 **QA洞察**：向量数据库选型需在召回率、延迟、成本之间权衡，没有"最好"只有"最适合"。 |
| 🔲 | **Day 29** | **[关键词检索基准：BM25、TF-IDF](./Phase3_RAG_Reliability/Day29)** | 💡 **QA洞察**：稀疏检索在精确匹配场景下仍有优势，与稠密检索融合可提升整体召回率10-20%。 |
| 🔲 | **Day 30** | **[混合检索架构：稀疏+密集融合](./Phase3_RAG_Reliability/Day30)** | 💡 **QA洞察**：混合检索的权重分配是核心挑战，需通过A/B测试找到最优融合策略。 |
| 🔲 | **Day 31** | **[重排序模型评估：排序准确性 vs 延迟](./Phase3_RAG_Reliability/Day31)** | 💡 **QA洞察**：重排序可显著提升Top-K准确率，但会增加延迟，需在质量与性能间找到平衡点。 |
| 🔲 | **Day 32** | **[检索层性能压测与容量规划](./Phase3_RAG_Reliability/Day32)** | 💡 **QA洞察**：检索层性能需关注P99延迟而非平均值，突发流量下的降级策略是生产必备。 |
| 🔲 | **Day 33** | **[检索质量监控与漂移检测](./Phase3_RAG_Reliability/Day33)** | 💡 **QA洞察**：Embedding漂移是渐进式的，需建立基线监控，及时发现数据分布变化。 |

### 🤖 第四阶段：工具链重构 —— Agent 智能体的测试落地 (Day 51-70)

*应对 Agent "状态空间爆炸"和"决策路径不可预测"的终极测试挑战。*

> 💡 **QA 洞察**：Agent 测试的核心挑战是**状态空间爆炸**和**决策路径不可预测**，需要新的测试范式。

## 📊 项目独家特色：AI 风险分级体系

在传统的 Bug 评级基础上，本项目探索建立了针对 AI 系统的**三级质量风控拦截标准**：

| 级别 | 定义 (AI 场景) | 门禁响应机制示例 |
|:---:|:---|:---|
| 🔴 **L1 阻断性** | 越狱攻击成功 / 产生违背法律法规的幻觉 / 关键功能完全失效 | 线上熔断，立即拦截输出并告警，阻断发布 |
| 🟡 **L2 高优先级** | Faithfulness（忠实度）低于 0.8 / 性能严重退化 / 安全边界模糊 | 降级处理，提示用户"此回答存在不确定性"，限期修复 |
| 🟢 **L3 一般风险** | 回答冗长 / 语气未能严格遵循 Prompt 人设 / 边缘场景处理不当 | 记录日志，进入下一次迭代优化池 |

### 质量门禁代码示例

```python
# 质量门禁检查器
L1_BLOCKERS = [
    "intent_accuracy_below_threshold",   # 意图识别 < 95%
    "symptom_recall_below_threshold",    # 症状召回 < 98%
    "hallucination_above_threshold",     # 幻觉率 > 1%
    "security_test_failed",              # 安全测试失败
    "emergency_intent_misclassified",    # 紧急意图误判
]

L2_WARNINGS = [
    "robustness_decay_above_threshold",  # 鲁棒性衰减 > 5%
    "p99_latency_above_threshold",       # P99延迟 > 2s
    "stability_variance_high",           # 稳定性方差过高
]
```
