AI 正在吃掉"点点点"和"写脚本"。但它吃不掉一件事：**判断一个系统（尤其是 AI 系统）到底靠不靠谱、敢不敢放行。**

> 市面上烂大街的教程都在教你"怎么调包写 AI 应用"；但几乎没人教你——**当大模型胡说八道时，如何精准定位 Bug？如何给 AI 系统建立质量风控体系？如何成为那个连 AI 都需要你来守门的人？**

## 🎭 为什么会有这个项目？

我是一名 43 岁、拥有十几年经验的传统自动化测试工程师。面对 AI 浪潮，我和很多同行一样焦虑过。这些都是测试群里真实的声音：

> "我们公司也在强推 AI，但写出来的用例还要我花时间擦屁股……"
> "AI 写的代码有问题，测试守门；那测试也 AI 了，**谁来守门？**"

我的答案是：**不去卷底层算法（那不是我的优势），而是把十几年练就的"质量直觉"升维**——从 **"系统稳定性、质量风控、缺陷注入、自动化评估"** 的 QA 视角重新解构 AI 系统，成为那个**守门的人**。

这个仓库记录了我从零跑通 AI 系统、并为它建立测试与质量体系的全过程。如果你和我一样有转型焦虑，欢迎 Fork 和 Star，我们一起破局。

## 📊 项目独家特色：AI 风险分级体系

在传统 Bug 评级基础上，本项目建立了针对 AI 系统的**三级质量风控拦截标准**（贯穿全部主题、并落地为 [05 质量门禁](./05_质量门禁CI)）：

| 级别 | 定义（AI 场景） | 门禁响应 |
|:---:|:---|:---|
| 🔴 **L1 阻断性** | 越狱成功 / 产生违法内容 / 幻觉率超标 / 敏感信息泄露 | 阻断发布，立即拦截并告警 |
| 🟡 **L2 高优先级** | 忠实度 < 0.8 / 性能严重退化 / 鲁棒性衰减 | 告警，提示不确定性，限期修复 |
| 🟢 **L3 一般风险** | 回答冗长 / 语气漂移 / 边缘场景处理不当 | 记录日志，进入优化池 |

## 🚀 快速开始

所有脚本都**开箱即用、极简依赖**，只暴露最核心的数据流和测试点。

```bash
# 1. 安装依赖
pip install -r requirements.txt

# 2. 配置真实模型（复制模板后填入你的 key）
cp .env.example .env
#   推荐 DeepSeek / 智谱等含免费额度的国产兼容端点

# 3. 运行任一主题（以幻觉检测为例）
pytest 01_幻觉检测/test_hallucination.py -v -s
```

> **诚实底线**：未配置真实 API Key 时，需要模型的测试会**直接跳过（skip）并说明原因**，而**绝不用假数据伪造"通过"**。一个教别人"抓 AI 幻觉"的项目，自己的测试结论必须是真的。
