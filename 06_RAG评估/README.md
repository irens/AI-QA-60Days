# 06 · RAG 评估：检索准不准 vs 答得对不对

> 测 RAG 不要当成测传统接口！必须把 **"检索准不准"** 和 **"答得对不对"** 这两个组件**解耦**来测。

---

## 🎯 守门点：解耦评估

RAG = 检索（Retrieval）+ 生成（Generation）。混在一起测，出了问题分不清责任：

| 故障 | 根因 | 解耦后才能定位 |
|---|---|---|
| 答案错了 | 可能是**没捞到**正确资料（检索的锅） | 检索命中率 |
| 答案错了 | 也可能是**捞到了却答错/没用上**（生成的锅） | 答案正确率 |

> 不解耦，就只能笼统说"RAG 不准"，无从下手优化。

---

## 🧪 核心方法

用一个极简知识库 + 朴素检索器，把评估拆成两层：

| 层 | 测什么 | 特性 |
|---|---|---|
| ① 检索层 | 是否捞到了**包含答案的正确文档**（命中率） | **确定性**，不需 API Key，永远可跑 |
| ② 生成层 | 基于捞到的资料，**答案是否正确** | 真实调用模型 |

> 检索层是确定性的（朴素 bigram 重叠检索）——这意味着这道关**任何时候都能跑、结果可复现**，不依赖外部模型。

---

## 📐 门禁

| 级别 | 触发 | 含义 |
|:--:|---|---|
| 🔴 L1 | 检索命中率 < 0.8 | 资料没捞对，生成再强也白搭 |
| 🟡 L2 | 答案正确率 < 0.8 | 捞对了却没答对，生成层有问题 |

---

## ▶️ 运行

```bash
pytest 06_RAG评估/test_rag_eval.py -v -s
```

> **诚实声明**：检索层永远可跑；生成层在未配置真实 API Key 时**跳过**，不伪造结果。
> 说明：本例检索器刻意做得很朴素（演示评估方法，而非检索算法）。生产中会换成向量检索 / 混合检索 / 重排序，但**解耦评估的思路不变**。
