# Day 51: 工具选择正确性测试

## 🎯 1. 核心风险与测试目标 (20分钟)
> **测试架构师视角**：Agent 选错工具就像医生开错药，后果可能是灾难性的——从数据泄露到业务损失。

### 业务风险点
| 风险场景 | 潜在损失 | 发生频率 |
|---------|---------|---------|
| 查询工具 vs 修改工具混淆 | 误删/误改生产数据 | 中 |
| 高权限工具被低权限请求触发 | 权限提升攻击 | 低 |
| 相似功能工具选择错误 | 返回错误信息，用户信任下降 | 高 |
| 多工具场景遗漏必要工具 | 任务未完成，用户体验差 | 中 |

**真实案例**：某客服 Agent 将"删除订单"意图识别为"查询订单"，导致用户无法取消误购商品，引发投诉。

### 测试思路
1. **意图-工具映射测试**：构建覆盖所有工具触发条件的测试集
2. **边界模糊测试**：设计意图表述模糊、多工具都可能适用的场景
3. **权限边界测试**：验证低权限请求无法触发高权限工具
4. **工具依赖测试**：验证多步骤任务中工具调用顺序正确性

## 📚 2. 核心理论

### 2.1 工具选择的三层决策

```
用户输入
    ↓
[意图识别层] → 理解用户想要什么
    ↓
[工具匹配层] → 从可用工具中选择最合适的
    ↓
[权限校验层] → 验证当前上下文是否有权限调用
    ↓
工具执行
```

### 2.2 工具描述的关键要素

工具选择依赖 LLM 对工具描述的理解，一个高质量的工具描述应包含：

| 要素 | 作用 | 测试关注点 |
|-----|------|-----------|
| `name` | 工具标识 | 命名是否清晰、无歧义 |
| `description` | 功能说明 | 是否准确描述用途和边界 |
| `parameters` | 参数定义 | schema 是否完整、类型是否明确 |
| `required` | 必填参数 | 是否与实际业务逻辑一致 |

### 2.3 常见的工具选择错误模式

1. **描述重叠**：两个工具功能描述相似，LLM 难以区分
2. **意图漂移**：用户输入包含多个意图，LLM 只识别了其中一个
3. **上下文缺失**：缺少对话历史，导致工具选择脱离上下文
4. **过度匹配**：工具描述包含热门关键词，被错误触发
