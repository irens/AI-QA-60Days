# Day 53: 结果解析失败处理测试

## 🎯 1. 核心风险与测试目标 (20分钟)
> **测试架构师视角**：工具调用成功只是开始，结果解析失败会让整个流程功亏一篑，且用户感知为"系统无响应"。

### 业务风险点
| 风险场景 | 潜在损失 | 用户体验影响 |
|---------|---------|-------------|
| 空结果未处理 | 用户看到空白回复 | 困惑、重复提问 |
| 异常结构解析失败 | 流程中断，无后续动作 | 任务未完成 |
| 超时结果丢弃 | 用户等待后无响应 | 强烈不满 |
| 部分失败未识别 | 错误数据进入下游 | 级联错误 |
| 错误信息暴露 | 敏感信息泄露 | 安全风险 |

**真实案例**：某天气查询 Agent 在 API 限流时返回空结果，Agent 未做处理直接回复"今天天气很好"，导致用户未带伞淋雨后在社交媒体投诉。

### 测试思路
1. **空结果处理**：工具返回空/None 时，Agent 的应对策略
2. **异常结构处理**：返回 JSON 与预期 Schema 不符时的容错
3. **超时处理**：工具调用超时后的降级策略
4. **部分失败处理**：批量操作中部分成功部分失败的场景
5. **错误信息脱敏**：确保不向用户暴露内部错误详情

## 📚 2. 核心理论

### 2.1 工具结果处理的完整链路

```
工具调用
    ↓
[结果接收] → 获取原始响应
    ↓
[格式解析] → JSON/XML/文本 → 结构化数据
    ↓
[有效性验证] → 检查空值、必填字段、数据范围
    ↓
[业务逻辑处理] → 提取关键信息、格式化输出
    ↓
[用户回复生成] → 自然语言回复
    ↓
[异常兜底] → 错误处理、降级策略
```

### 2.2 结果解析失败的分类

| 失败类型 | 触发条件 | 处理策略 |
|---------|---------|---------|
| **格式错误** | 返回非预期格式（HTML 而非 JSON） | 尝试多种解析器，记录日志 |
| **结构缺失** | 关键字段缺失 | 使用默认值，标记不完整 |
| **数据异常** | 空数组、null 值、超范围数据 | 空值检测，边界检查 |
| **超时失败** | 工具响应超过阈值 | 返回超时提示，提供重试 |
| **部分失败** | 批量操作部分成功 | 区分成功/失败项，分别处理 |

### 2.3 降级策略设计原则

```python
# 降级策略层级
def handle_tool_result(result, context):
    # L1: 正常处理
    if result.is_valid():
        return process_success(result)
    
    # L2: 部分成功处理
    if result.is_partial():
        return process_partial(result)
    
    # L3: 缓存回退
    if cache.has(context):
        return cache.get(context) + " (缓存数据)"
    
    # L4: 默认值
    if context.has_default():
        return context.default_response
    
    # L5: 人工转接
    return escalate_to_human(context)
```
