# Day 65: 负载测试与容量规划

## 🎯 1. 核心风险与测试目标 (20分钟)

> **测试工程师视角**：生产环境容量不足导致服务降级，无法支撑业务增长。如果不进行充分测试，线上会出现：
> - 大促流量突增导致服务崩溃
> - 容量评估不准确造成资源浪费或不足
> - 性能瓶颈定位困难，优化方向不明确
> - 缺乏弹性伸缩策略，无法应对流量波动

### 1.1 业务风险点

| 风险等级 | 风险场景 | 业务影响 |
|---------|---------|---------|
| 🔴 L1 阻断性 | 容量不足导致服务完全不可用 | 业务停摆，收入归零 |
| 🔴 L1 阻断性 | 性能拐点未识别，突发流量压垮系统 | 用户体验极差，品牌受损 |
| 🟡 L2 高优先级 | 容量规划过度，资源浪费 | 成本增加，ROI下降 |
| 🟡 L2 高优先级 | 瓶颈定位错误，优化无效 | 投入产出比低 |
| 🟢 L3 一般 | 监控告警阈值设置不当 | 误报或漏报 |

### 1.2 测试思路

**核心策略**：通过负载测试找到系统性能拐点，为容量规划提供数据支撑

1. **基准测试**：确定系统在正常负载下的性能基线
2. **负载测试**：验证系统在预期负载下的表现
3. **压力测试**：找到系统性能拐点和崩溃点
4. **稳定性测试**：验证系统长时间运行的稳定性
5. **容量规划**：基于测试数据制定扩容策略

## 📚 2. 理论基础

### 2.1 性能测试类型

```
┌─────────────────────────────────────────────────────────────┐
│                   性能测试金字塔                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                    ┌─────────────┐                         │
│                    │  压力测试    │  找到崩溃点             │
│                    │  (Spike)    │  验证恢复能力            │
│                    └──────┬──────┘                         │
│                           │                                 │
│              ┌────────────┴────────────┐                   │
│              │       负载测试           │  验证预期负载       │
│              │     (Load Test)         │  确定性能拐点       │
│              └────────────┬────────────┘                   │
│                           │                                 │
│        ┌──────────────────┴──────────────────┐             │
│        │            基准测试                  │  建立基线      │
│        │          (Benchmark)                │  对比优化效果   │
│        └──────────────────┬──────────────────┘             │
│                           │                                 │
│  ┌────────────────────────┴────────────────────────┐       │
│  │                  稳定性测试                      │  长时间运行   │
│  │               (Soak Test)                       │  发现内存泄漏  │
│  └─────────────────────────────────────────────────┘       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 关键性能指标

| 指标 | 说明 | 健康阈值 | 测试方法 |
|-----|------|---------|---------|
| TPS/QPS | 每秒事务/查询数 | 按业务需求 | 压力测试 |
| 响应时间 | P50/P95/P99 | P95<200ms | 负载测试 |
| 错误率 | 失败请求占比 | <0.1% | 全阶段 |
| CPU使用率 | 系统负载 | <70% | 监控 |
| 内存使用 | 堆内存占用 | <80% | 监控 |
| 连接数 | 活跃连接数 | <最大连接80% | 连接池监控 |

### 2.3 容量规划方法

| 方法 | 适用场景 | 计算公式 |
|-----|---------|---------|
| 峰值倍数法 | 流量波动大 | 容量 = 峰值 × 安全系数(1.5-2) |
| 线性外推法 | 增长稳定 | 容量 = 当前 × (1 + 增长率)^n |
| 排队论模型 | 精确计算 | 基于Erlang公式 |
| 历史数据分析 | 有历史数据 | 基于同期数据预测 |
