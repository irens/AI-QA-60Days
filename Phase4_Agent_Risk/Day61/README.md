# Day 61: 备用路径切换机制测试

## 🎯 1. 核心风险与测试目标 (20分钟)

> **测试架构师视角**：当主路径熔断或不可用时，备用路径能否无缝接管？切换过程是否会导致数据丢失或状态不一致？

### 1.1 业务风险点

| 风险等级 | 风险场景 | 业务影响 |
|---------|---------|---------|
| 🔴 L1 阻断性 | 备用路径切换失败，服务完全不可用 | 业务停摆，用户体验归零 |
| � L1 阻断性 | 切换过程中数据丢失或重复处理 | 数据不一致，业务逻辑错误 |
| �🟡 L2 高优先级 | 备用路径性能远低于主路径 | 用户体验严重下降 |
| � L2 高优先级 | 主备路径切换过于频繁（抖动） | 系统不稳定，资源浪费 |
| �🟢 L3 一般 | 切换延迟过长 | 短暂服务中断 |

### 1.2 测试思路

**核心策略**：验证故障场景下备用路径的可用性和切换的可靠性

1. **切换触发测试**：验证各种故障条件下是否正确触发切换
2. **数据一致性测试**：验证切换过程中数据不丢失、不重复
3. **回切测试**：验证主路径恢复后能否正确回切
4. **抖动防护测试**：验证防止频繁切换的机制

## 📚 2. 核心理论

### 2.1 主备路径架构

```
┌─────────────────────────────────────────────────────────────────┐
│                     主备路径切换架构                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   用户请求 ──> ┌─────────────┐    健康        ┌─────────────┐   │
│                │   路由器    │ ────────────> │  主路径      │   │
│                │  (Router)   │               │ (Primary)   │   │
│                └──────┬──────┘               └─────────────┘   │
│                       │                                         │
│                   不健康│ 触发切换                               │
│                       ▼                                         │
│                ┌─────────────┐               ┌─────────────┐   │
│                │   健康检查   │ ───────────> │  备用路径    │   │
│                │  (Health)   │               │ (Fallback)  │   │
│                └─────────────┘               └─────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 切换策略类型

| 策略 | 说明 | 适用场景 |
|-----|------|---------|
| **故障切换** (Failover) | 主路径故障后切换到备用 | 高可用要求场景 |
| **负载均衡** (Load Balance) | 多路径同时工作，故障时剔除 | 高性能要求场景 |
| **降级切换** (Degrade) | 切换到简化版服务 | 资源受限场景 |
| **熔断切换** (Circuit) | 熔断触发后切换 | 防止级联故障 |

### 2.3 LangChain中的路径切换场景

```python
# 场景1: LLM Provider 切换
Primary: OpenAI GPT-4 → Fallback: Azure OpenAI → Fallback: Local LLM

# 场景2: 检索策略切换
Primary: 向量检索 → Fallback: 关键词检索 → Fallback: 全量扫描

# 场景3: 工具调用切换
Primary: 实时API → Fallback: 缓存数据 → Fallback: 静态默认值

# 场景4: 解析器切换
Primary: StructuredOutputParser → Fallback: PydanticOutputParser → Fallback: 正则提取
```
