# Day 67: 数据隔离与隐私合规测试

## 🎯 1. 核心风险与测试目标 (20分钟)

> **测试工程师视角**：多用户数据混淆、敏感信息泄露、合规违规。如果不进行充分测试，线上会出现：
> - 用户A看到用户B的私密数据，信任崩塌
> - PII信息泄露，面临法律诉讼和巨额罚款
> - 不合规操作导致业务被叫停

### 1.1 业务风险点

| 风险等级 | 风险场景 | 业务影响 |
|---------|---------|---------|
| 🔴 L1 阻断性 | 用户数据隔离失效，跨用户数据泄露 | 用户信任崩塌，品牌受损 |
| 🔴 L1 阻断性 | 租户间数据交叉污染 | 企业客户流失，合同违约 |
| 🔴 L1 阻断性 | PII敏感信息未识别/未脱敏 | 法律风险，GDPR/CCPA罚款 |
| 🔴 L1 阻断性 | 未授权访问未拦截 | 数据泄露，安全事件 |
| 🟡 L2 高优先级 | 数据保留策略未执行 | 合规风险，存储成本增加 |
| 🟡 L2 高优先级 | GDPR权利（导出/删除）不支持 | 违反法规，用户投诉 |
| 🟡 L2 高优先级 | 审计日志缺失 | 无法追溯，合规审计失败 |
| 🟢 L3 一般 | 传输/存储加密缺失 | 中间人攻击，数据窃取 |
| 🟢 L3 一般 | 权限控制过于宽松 | 内部威胁，数据滥用 |

### 1.2 测试思路

**核心策略**：模拟各种数据泄露场景，验证数据隔离和隐私保护机制

1. **隔离边界测试**：验证用户间、租户间的数据隔离
2. **PII检测**：测试敏感信息识别准确率
3. **脱敏验证**：确保敏感信息被正确脱敏
4. **合规检查**：验证GDPR/CCPA等法规要求
5. **审计追踪**：确认所有访问被记录

## 📚 2. 理论基础

### 2.1 数据分类等级

```
┌─────────────────────────────────────────────────────────────┐
│                      数据分类金字塔                           │
├─────────────────────────────────────────────────────────────┤
│  🔴 RESTRICTED (受限)     │  PII、财务数据、密码              │
│  🟡 CONFIDENTIAL (机密)   │  商业机密、用户偏好               │
│  🟠 INTERNAL (内部)       │  系统日志、运营数据               │
│  🟢 PUBLIC (公开)         │  公开文档、帮助信息               │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 多租户隔离模型

| 模型 | 隔离级别 | 适用场景 |
|-----|---------|---------|
| 独立数据库 | 物理隔离 | 高安全要求，高成本 |
| 共享数据库独立Schema | 逻辑隔离 | 平衡安全与成本 |
| 共享Schema行级隔离 | 租户ID隔离 | 成本敏感，SaaS |

### 2.3 隐私法规要求

**GDPR（欧盟）**：
- 数据可携带权（Right to data portability）
- 被遗忘权（Right to be forgotten）
- 数据最小化原则
- 隐私设计（Privacy by Design）

**CCPA（加州）**：
- 知情权（Right to know）
- 删除权（Right to delete）
- 选择退出权（Right to opt-out）

**PIPL（中国）**：
- 告知-同意原则
- 敏感个人信息特别保护
- 数据本地化要求

### 2.4 敏感信息类型

- **PII（个人身份信息）**：姓名、身份证号、手机号、邮箱
- **财务信息**：银行卡号、交易记录
- **凭证信息**：密码、API密钥、Token
- **生物识别**：指纹、面部特征
