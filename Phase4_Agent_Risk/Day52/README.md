# Day 52: 参数格式解析测试

## 🎯 1. 核心风险与测试目标 (20分钟)
> **测试架构师视角**：参数解析错误是 Agent 系统的"静默杀手"——它不会崩溃，但会产生错误结果，且难以发现。

### 业务风险点
| 风险场景 | 潜在损失 | 检测难度 |
|---------|---------|---------|
| 类型转换失败 | 工具调用异常，流程中断 | 低 |
| 必填参数缺失 | 工具返回错误或不完整结果 | 中 |
| 参数格式错误 | 数据写入错误，产生脏数据 | 高 |
| 边界值溢出 | 系统异常或安全漏洞 | 高 |
| 特殊字符注入 | SQL/命令注入攻击 | 高 |

**真实案例**：某订单系统因日期格式解析错误（YYYY-MM-DD vs DD/MM/YYYY），导致促销活动期间大量订单时间戳错误，库存同步失败。

### 测试思路
1. **Schema 合规性测试**：验证 LLM 输出是否符合 JSON Schema
2. **类型边界测试**：数字溢出、字符串长度、数组边界
3. **必填参数验证**：故意省略必填项，测试系统响应
4. **格式兼容性测试**：日期、时间、货币等格式的多种表达方式
5. **注入攻击测试**：尝试在参数中注入恶意代码

## 📚 2. 核心理论

### 2.1 参数解析的完整链路

```
用户输入
    ↓
[意图理解] → 提取参数值
    ↓
[类型转换] → 字符串 → 目标类型
    ↓
[Schema验证] → 检查必填项、类型、范围
    ↓
[业务校验] → 检查业务规则（如日期不能是未来）
    ↓
工具调用
```

### 2.2 JSON Schema 验证要点

```json
{
  "type": "object",
  "properties": {
    "order_id": {
      "type": "string",
      "pattern": "^ORD-[0-9]{6}$",
      "description": "订单ID格式：ORD-六位数字"
    },
    "amount": {
      "type": "number",
      "minimum": 0.01,
      "maximum": 100000,
      "description": "金额范围：0.01-100000"
    },
    "quantity": {
      "type": "integer",
      "minimum": 1,
      "maximum": 999,
      "description": "数量：1-999的整数"
    }
  },
  "required": ["order_id", "amount"]
}
```

### 2.3 常见的参数解析错误

| 错误类型 | 示例 | 后果 |
|---------|------|------|
| 类型混淆 | `"123"` vs `123` | 工具接收错误类型，执行失败 |
| 格式不一致 | `"2024-01-01"` vs `"01/01/2024"` | 日期解析错误 |
| 编码问题 | 中文参数被 URL 编码 | 显示乱码或匹配失败 |
| 精度丢失 | 浮点数 `19.99` 变成 `19.9899999999` | 金额计算错误 |
| 空值处理 | `null` vs `""` vs 未提供 | 逻辑分支错误 |
