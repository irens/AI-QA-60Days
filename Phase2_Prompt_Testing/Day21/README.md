# Day 21: Prompt版本管理与A/B测试

## 🎯 1. 核心风险与测试目标 (20分钟)
> **测试架构师视角**：Prompt变更是AI系统的"代码变更"，没有版本管理就像没有Git的软件开发，风险不可控。

### 业务风险点
- **未经测试的Prompt上线**：直接修改生产Prompt可能导致服务质量骤降，用户投诉激增
- **无法回滚**：发现Prompt问题后无法快速恢复到稳定版本，业务受损时间延长
- **效果不可比**：多个Prompt版本并行时，无法科学评估哪个版本更优，决策凭感觉
- **变更不可追溯**：无法追踪哪个Prompt版本引入了问题，故障排查困难

### 测试思路
- **版本控制测试**：验证Prompt版本管理的完整性和一致性
- **A/B效果对比**：科学评估不同Prompt版本的效果差异
- **回滚机制验证**：测试紧急情况下Prompt回滚的速度和准确性
- **变更影响分析**：评估Prompt变更对系统行为的潜在影响范围

### 2.3 A/B测试核心要素

**统计显著性检验**：
```
假设：
H0: 版本A和版本B的效果无差异 (μA = μB)
H1: 版本A和版本B的效果有差异 (μA ≠ μB)

检验统计量：
z = (pA - pB) / sqrt(p*(1-p)*(1/nA + 1/nB))

其中：
- pA, pB: 两个版本的转化率/准确率
- nA, nB: 样本量
- p: 合并转化率

决策规则：
- |z| > 1.96 (α=0.05): 拒绝H0，差异显著
- |z| <= 1.96: 无法拒绝H0，差异不显著
```

**早期停止规则**：
```python
def should_stop_early(control_metrics, treatment_metrics, min_samples=1000):
    """
    早期停止判断
    - 达到最小样本量后才判断
    - 效果差异超过阈值时停止
    - 运行时间过长时停止
    """
    if control_metrics['n'] < min_samples or treatment_metrics['n'] < min_samples:
        return False, "样本量不足"
    
    # 计算置信区间
    lift = (treatment_metrics['mean'] - control_metrics['mean']) / control_metrics['mean']
    
    # 如果效果提升超过20%或下降超过10%，提前停止
    if lift > 0.20:
        return True, f"效果显著提升: {lift:.1%}"
    elif lift < -0.10:
        return True, f"效果显著下降: {lift:.1%}"
    
    return False, "继续观察"
```
