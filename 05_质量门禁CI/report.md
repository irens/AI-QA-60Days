# 质量报告 · 05 质量门禁接进 CI/CD

> 本主题为**确定性逻辑**，结果可复现，无需 API Key。

---

## 一、测试目标

验证质量门禁引擎的核心契约：
- 干净指标 → 放行（exit 0）
- L1 命中 → **必阻断**（exit 1）
- 仅 L2 命中 → 告警但放行
- L1 与 L2 并存 → L1 主导，阻断
- 未提供的指标 → 不误判

## 三、命令行门禁演示（真实输出）

输入一组演示指标（幻觉率 3%、忠实度 0.72、冗余度 0.41）：

```
🛡️  质量门禁报告
🔴 L1 阻断：
   - 幻觉率 > 1%（实测 hallucination_rate=0.03）
🟡 L2 告警：
   - 忠实度 < 0.8（实测 faithfulness=0.72）
🟢 L3 记录：
   - 回答冗长（冗余度 > 0.3）（实测 answer_verbosity=0.41）
结论：⛔ 阻断发布（BLOCKED）  | 退出码=1
```

**退出码 1** 正是 CI 流水线的"拦截开关"——GitHub Actions 据此让 PR 检查失败，变更合不进去。

## 四、守门人洞察

> 把"质量好不好"从**一句主观评价**，变成**一个可执行、可量化、可审计的退出码**。

- **退出码就是权力**：`exit 1` 让质量标准第一次有了"硬约束"——不达标，机器替你拦住，不靠人盯、不靠自觉。
- **三级分层是关键**：不是所有问题都该卡发布。L1 阻断（安全/事实硬伤），L2 告警放行（质量打折但可控），L3 仅记录（体验瑕疵）。一刀切只会让门禁要么形同虚设、要么寸步难行。
- **规则与代码分离**：阈值集中在规则表里，可版本管理、可按业务调整、可在评审中讨论——这本身就是"质量标准工程化"。

这一篇把前面 01、09 的单点检测，升格成了一道**真正能挡住坏变更的发布卡口**。

---

## 五、局限与下一步

- 本引擎消费的是"已经算好的指标"；指标怎么来，靠 01/09 等检测产出 `metrics.json`，需在 CI 里串成完整流水线。
- 阈值（1%、0.8、2s 等）是示例值，**生产必须按业务校准**——错误的阈值比没有门禁更危险（要么漏放、要么误杀）。
