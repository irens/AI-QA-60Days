# Day 40: 上下文信息利用不足检测

## 🎯 1. 核心风险与测试目标 (20分钟)
> **测试架构师视角**：RAG系统检索了大量相关文档，但LLM只利用了其中一小部分，导致答案不完整或遗漏关键信息。

### 业务风险点
- **答案不完整**：用户问题涉及多个方面，但答案只回应了部分内容
- **关键信息遗漏**：检索文档中包含重要警告、限制条件，但答案未提及
- **决策失误**：基于不完整的答案做出错误业务决策，如医疗建议遗漏副作用说明
- **合规风险**：金融/法律场景下，遗漏关键条款可能导致法律责任

### 测试思路
1. **引用覆盖率分析**：检测答案中引用了多少检索到的文档片段
2. **信息熵分析**：评估答案的信息量与检索文档信息量的比例
3. **多文档问题测试**：设计需要综合多个文档才能回答完整的问题
4. **关键信息定位**：在检索文档中植入关键信息，检测是否被答案覆盖

## 📚 2. 核心理论

### 2.1 上下文利用不足的定义

**Context Underutilization** 指LLM未能充分利用提供给它的上下文信息，表现为：
- 只引用了部分检索文档
- 忽略了文档中的重要细节
- 未能综合多个文档的信息

### 2.2 根因分析

| 根因 | 说明 | 测试关注点 |
|-----|------|-----------|
| 注意力分散 | 上下文过长，关键信息被"淹没" | 长上下文下的关键信息定位能力 |
| 位置偏见 | LLM对上下文中间位置的信息关注度低 | 关键信息放置位置的影响 |
| 信息冲突 | 多个文档信息矛盾时，模型选择忽略 | 冲突信息的处理能力 |
| 生成惯性 | 模型生成答案时"偷懒"，不深入检索文档 | 答案长度与信息量的关系 |

### 2.3 评估指标

```
上下文利用率 = 答案中引用的有效信息片段数 / 检索提供的总信息片段数

信息覆盖率 = 答案覆盖的关键信息点 / 问题所需的关键信息点

引用完整度 = 实际引用的文档数 / 应该引用的文档数
```

### 2.4 测试方法

**多跳问题测试**：
- 问题需要综合2-3个文档的信息才能完整回答
- 检测模型是否能"连接"多个文档的信息

**关键信息植入测试**：
- 在检索文档的不同位置植入特定信息
- 检测答案是否包含这些关键信息

**冗余信息干扰测试**：
- 在上下文中加入相关但非必要的文档
- 检测模型是否能识别并聚焦核心信息

### 测试场景预览

   - 在相同长度上下文中放置不同密度的关键信息
   - 检测模型在高密度信息下的利用能力
