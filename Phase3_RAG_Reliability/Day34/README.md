# Day 34: 忠实度自动化检测 - NLI Entailment方法

## 🎯 1. 核心风险与测试目标 (20分钟)
> **测试架构师视角**：LLM生成的答案可能"一本正经地胡说八道"，忠实度检测是RAG系统的最后一道防线。

### 业务风险点
- **幻觉传播**：LLM生成与上下文不符的内容，导致用户基于错误信息做决策
- **事实污染**：部分真实信息中掺杂虚假内容，难以人工识别
- **信任崩塌**：一旦用户发现答案不可信，将永久失去对系统的信任

### 测试思路
基于**自然语言推理（NLI）**的Entailment检测方法：
1. 将答案拆分为原子声明（Atomic Claims）
2. 逐一验证每个声明是否能被上下文支持
3. 计算忠实度分数 = 被支持的声明数 / 总声明数

## 📚 2. 核心理论

### 2.1 NLI三分类框架

| 关系类型 | 定义 | 示例 |
|---------|------|------|
| **Entailment（蕴含）** | 前提支持假设 | 前提："北京是中国的首都" → 假设："北京在中国" |
| **Contradiction（矛盾）** | 前提否定假设 | 前提："今天下雨" → 假设："今天是晴天" |
| **Neutral（中性）** | 无法确定关系 | 前提："他喜欢吃苹果" → 假设："他喜欢水果" |

### 2.2 Ragas Faithfulness计算流程

```
┌─────────────┐    ┌──────────────┐    ┌─────────────┐    ┌─────────────┐
│   Answer    │───→│ Claim        │───→│ NLI Verify  │───→│  Score      │
│   答案文本   │    │ Extraction   │    │ per Claim   │    │  Calc       │
└─────────────┘    └──────────────┘    └─────────────┘    └─────────────┘
      │                  │                   │                  │
      ▼                  ▼                   ▼                  ▼
 "北京人口2180万"    [Claim1, Claim2]   Entailment:2      2/3 = 0.67
 "上海是金融中心"                       Contradiction:1
 "深圳在广东省"                         Neutral:0
```

### 2.3 关键阈值设定

| 忠实度分数 | 风险等级 | 处理建议 |
|-----------|---------|---------|
| 0.9 - 1.0 | L3（低） | 直接输出，无需干预 |
| 0.7 - 0.9 | L2（中） | 添加免责声明或人工审核 |
| 0.0 - 0.7 | L1（高） | 拒绝回答或重新检索 |
