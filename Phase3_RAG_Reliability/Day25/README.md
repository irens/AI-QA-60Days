# Day 25: 固定长度分块策略评估（基础篇）

## 🎯 1. 核心风险与测试目标 (20分钟)
> **测试架构师视角**：固定长度分块看似简单，但边界信息丢失和语义断裂是RAG召回率低的隐形杀手。

### 业务风险点
| 风险场景 | 线上事故 | 影响程度 |
|---------|---------|---------|
| 句子被截断 | 用户查询"退款政策"，但关键信息"7天内"被切到下一个chunk，导致回答不完整 | 🔴 高 |
| 重叠不足 | 上下文依赖的段落被分散，检索时无法获取完整逻辑 | 🟡 中 |
| 块大小不当 | 块太大导致噪声多，块太小导致信息碎片化 | 🟡 中 |
| 编码边界错误 | 多字节字符被截断导致乱码或解析失败 | 🔴 高 |

### 测试思路
1. **边界探测**：在句子中间、段落边界、标点符号处切割，验证语义完整性
2. **重叠验证**：测试不同重叠比例（0%、10%、20%、50%）对信息连续性的影响
3. **块大小扫描**：从100到2000 token，找出最佳召回率区间
4. **特殊字符**：测试中文、emoji、代码块等边界处理

## 📚 2. 核心理论

### 2.1 固定长度分块原理
```
文档 → [chunk1][chunk2][chunk3]...
       每个chunk固定token数（如500）
       可选：重叠部分（如50 token）
```

**核心参数**：
- `chunk_size`: 每个块的最大token数
- `chunk_overlap`: 相邻块之间的重叠token数

### 2.2 关键问题分析

#### 问题1：语义断裂
```
原文："用户可以在7天内申请退款，但需要提供购买凭证。"
分块结果：
- Chunk1: "用户可以在7天内申"  ← 信息不完整
- Chunk2: "请退款，但需要提供购买凭证。"  ← 缺少时间条件
```

#### 问题2：重叠策略的权衡
| 重叠比例 | 优点 | 缺点 |
|---------|------|------|
| 0% | 存储效率高 | 边界信息丢失风险 |
| 10-20% | 平衡方案 | 适度冗余 |
| 50%+ | 信息完整性好 | 存储成本翻倍 |

### 2.3 行业最佳实践
- **LangChain默认**：chunk_size=1000, chunk_overlap=200
- **推荐起点**：chunk_size=500-1000, chunk_overlap=10-20%
- **中文场景**：建议按字符数而非token数，避免编码问题

## 📖 扩展阅读

### 固定长度分块适用场景
| 场景 | 适用性 | 原因 |
|-----|-------|------|
| 结构化文档（表格、列表） | ⭐⭐⭐⭐⭐ | 格式规整，边界清晰 |
| 代码文档 | ⭐⭐⭐⭐ | 函数/类边界明确 |
| 长篇文章 | ⭐⭐⭐ | 需要足够重叠保证连贯性 |
| 对话记录 | ⭐⭐ | 上下文依赖强，易断裂 |
| 技术手册 | ⭐⭐⭐⭐ | 章节结构清晰 |

### 常见陷阱
1. **盲目使用默认参数**：不同文档类型需要不同的块大小
2. **忽略重叠设置**：0%重叠在大多数情况下是危险的
3. **不考虑查询长度**：块大小应与预期查询长度匹配
4. **缺乏召回率验证**：没有量化指标支撑参数选择
