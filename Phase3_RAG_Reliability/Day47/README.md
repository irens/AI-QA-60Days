# Day 47: LLM-as-a-Judge 评估Prompt设计与评分标准

## 🎯 1. 核心风险与测试目标 (20分钟)
> **测试架构师视角**：Prompt设计缺陷会导致评分标准漂移，让评估结果失去可比性。

### 业务风险点
| 风险场景 | 后果 |
|---------|------|
| 评分标准模糊 | 评委理解不一致，同一样本多次评分差异大 |
| 评价维度缺失 | 遗漏关键质量维度，评估不全面 |
| 输出格式不稳定 | 无法自动解析评分结果，流水线中断 |
| Prompt注入风险 | 恶意输入操纵评委评分 |

### 测试思路
1. **评分标准清晰度测试**：验证不同评委对同一标准的理解一致性
2. **维度完整性测试**：检查评估维度是否覆盖业务需求
3. **输出格式稳定性测试**：验证评分结果的可解析性
4. **Prompt鲁棒性测试**：检测异常输入对评分的影响

## 📚 2. 核心理论

### 2.1 评估Prompt设计原则

```
┌─────────────────────────────────────────────────────────────┐
│                  高质量评估Prompt结构                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 角色定义 (Role)                                          │
│     "你是一位专业的内容质量评估专家..."                          │
│                                                             │
│  2. 任务描述 (Task)                                          │
│     "请评估以下回答的质量..."                                 │
│                                                             │
│  3. 评分标准 (Criteria)                                       │
│     - 准确性 (0-10分): 事实正确性...                          │
│     - 完整性 (0-10分): 信息覆盖度...                          │
│     - 清晰度 (0-10分): 表达易读性...                          │
│                                                             │
│  4. 评分细则 (Rubric)                                        │
│     10分: 完美，无改进空间                                     │
│     7-9分: 良好， minor issues                               │
│     4-6分: 一般，需要改进                                      │
│     1-3分: 较差， major issues                               │
│                                                             │
│  5. 输出格式 (Output Format)                                  │
│     ```json                                                  │
│     {"accuracy": 8, "completeness": 7, ...}                  │
│     ```                                                      │
│                                                             │
│  6. 示例 (Few-shot)                                          │
│     输入: ... → 评分: ...                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 常见评分标准维度

| 维度 | 定义 | 适用场景 | 权重建议 |
|-----|------|---------|---------|
| 准确性 | 信息的事实正确性 | 知识问答、事实核查 | 30-40% |
| 完整性 | 回答问题的全面程度 | 开放式问答 | 20-30% |
| 相关性 | 回答与问题的匹配度 | 检索增强生成 | 20-25% |
| 清晰度 | 表达的易读性和逻辑性 | 所有场景 | 10-15% |
| 安全性 | 是否包含有害内容 | 用户交互场景 | 15-20% |
| 创造性 | 回答的新颖性和独创性 | 创意写作 | 10-15% |

### 2.3 评分标准设计最佳实践

**1. 使用具体可观察的标准**
```
❌ 差："回答应该很好"
✅ 好："回答包含至少3个关键要点，每个要点都有具体例子支撑"
```

**2. 提供明确的评分锚点**
```
❌ 差："准确性: 1-10分"
✅ 好："准确性评分标准：
   - 10分: 所有事实准确，有权威来源支撑
   - 7-9分: 主要事实准确，minor inaccuracies
   - 4-6分: 部分事实错误或缺失
   - 1-3分: 主要事实错误或误导性"
```

**3. 多维度独立评分**
```python
# 推荐：各维度独立评分后加权
final_score = (
    accuracy * 0.35 +
    completeness * 0.25 +
    relevance * 0.20 +
    clarity * 0.10 +
    safety * 0.10
)
```

### 2.4 输出格式稳定性策略

| 策略 | 实现方式 | 优势 | 注意事项 |
|-----|---------|------|---------|
| JSON模式 | 强制JSON输出 | 结构化、易解析 | 需要模型支持 |
| 正则提取 | 从文本提取分数 | 兼容性好 | 需要验证完整性 |
| XML标签 | 使用特定标签包裹 | 清晰明确 | 增加token消耗 |
| 分隔符 | 使用固定分隔符 | 简单直接 | 需要处理边界情况 |
