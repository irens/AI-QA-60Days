# Day 33: 重排序优化与混合架构实战测试

## 🎯 1. 核心风险与测试目标 (20分钟)
> **测试架构师视角**：重排序是混合检索的"最后一公里"，排序错误将直接导致用户看到不相关的结果，前3个结果的准确性决定了80%的用户满意度。

### 业务风险点
| 风险场景 | 线上事故 | 影响程度 |
|---------|---------|---------|
| 重排序模型失效 | 相关文档排在第10位，用户看不到 | 🔴 高 |
| 延迟过高 | 重排序增加>500ms延迟，超时 | 🔴 高 |
| 上下文窗口溢出 | 输入超长导致截断，丢失关键信息 | 🟡 中 |
| 位置偏见 | 模型过度依赖原始排序位置 | 🟡 中 |
| 领域错配 | 通用重排序模型在专业领域效果差 | 🟡 中 |

### 测试思路
1. **重排序效果测试**：验证重排序能否提升Top-K准确性
2. **延迟开销测试**：测量重排序增加的延迟是否在可接受范围
3. **混合架构端到端测试**：验证完整流程的召回率和准确性
4. **异常场景测试**：超长文档、空结果、重复文档的处理
5. **不同重排序策略对比**：Cross-Encoder vs LLM-based

## 📚 2. 核心理论

### 2.1 混合检索架构完整流程
```
用户查询
    ↓
┌─────────────────────────────────────────────────────────────┐
│  第一阶段：召回（Recall）                                      │
│  ├── 关键词检索 → Top-K1 (稀疏检索)                           │
│  └── 语义检索 → Top-K2 (密集检索)                             │
│  └── 合并去重 → 候选集 (K1 + K2)                             │
└─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────────┐
│  第二阶段：重排序（Reranking）                                │
│  └── 重排序模型评分 → 精细化相关性排序                        │
│  └── 输出最终 Top-N                                          │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 重排序模型类型对比
| 类型 | 原理 | 优点 | 缺点 | 适用场景 |
|-----|------|-----|------|---------|
| Cross-Encoder | 拼接Query+Doc输入模型 | 准确率高 | 计算慢 | 候选集<100 |
| LLM-based | 使用大模型判断相关性 | 理解力强 | 成本高 | 高精度需求 |
| 两阶段 | Bi-Encoder初筛+CE精排 | 平衡 | 复杂 | 大规模检索 |

### 2.3 RRF (Reciprocal Rank Fusion) 融合算法
```python
def rrf_fusion(results_list, k=60):
    scores = defaultdict(float)
    for results in results_list:
        for rank, doc_id in enumerate(results):
            scores[doc_id] += 1.0 / (k + rank + 1)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
```
- **k值选择**：k越大，对原始排序位置越不敏感
- **推荐值**：k=60（经验值）

### 2.4 性能优化策略
| 策略 | 实现方式 | 效果 |
|-----|---------|------|
| 候选集截断 | 只重排序Top-100 | 减少90%计算 |
| 批量推理 | 一次处理多个Query-Doc对 | 提升GPU利用率 |
| 缓存 | 缓存热门查询结果 | 减少重复计算 |
| 异步更新 | 后台更新重排序分数 | 不影响主流程 |

## 📖 延伸阅读

### 重排序模型选型指南

| 场景 | 推荐方案 | 原因 |
|-----|---------|------|
| 高精度需求 | Cross-Encoder (如bge-reranker) | 准确率最高 |
| 成本敏感 | 轻量级CE模型 | 推理快 |
| 通用场景 | LLM-based (如GPT-4) | 理解力强 |
| 实时场景 | 两阶段架构 | 平衡效果和延迟 |

### 混合检索权重调优公式

```python
# 基础融合
final_score = α * keyword_score + β * semantic_score

# RRF融合（推荐）
final_score = 1/(k + keyword_rank) + 1/(k + semantic_rank)

# 加权RRF
final_score = w1 * 1/(k + keyword_rank) + w2 * 1/(k + semantic_rank)

# 典型配置
α = 0.3, β = 0.7  # 语义检索权重更高
k = 60            # RRF参数
w1 = 0.4, w2 = 0.6 # 加权RRF
```

### 生产环境最佳实践

1. **分层检索架构**
   ```
   L1: 关键词检索 (<5ms) → 快速过滤
   L2: 语义检索 (50-100ms) → 召回扩展
   L3: 重排序 (100-500ms) → 精确排序
   ```

2. **降级策略**
   ```python
   if reranker_latency > timeout:
       # 降级为RRF融合结果
       return rrf_results
   ```

3. **监控指标**
   - 召回率@K：候选集质量
   - NDCG@K：排序质量
   - 延迟P99：性能稳定性
   - 重排序命中率：模型有效性
