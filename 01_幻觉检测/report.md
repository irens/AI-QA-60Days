# 质量报告 · 01 幻觉检测（忠实度核验）

> 本报告基于**真实模型运行**（deepseek-chat），非模拟数据。

---

## 一、测试目标

验证一个"忠实度裁判"能否在给定上下文下，正确区分**忠实回答**与**幻觉回答**，并能否作为质量门禁使用（准确率门槛 ≥ 80%）。

- 测试对象：LLM-as-a-Judge 忠实度核验
- 裁判模型：`deepseek-chat`
- 样本：4 条（2 忠实 / 2 幻觉），通用领域（虚构产品"蓝鲸笔记 Pro"），零业务敏感

---

## 二、测试结果：✅ 通过（准确率 100%，4/4）

| 用例 | 期望 | 裁判判定 | 结果 |
|---|:--:|:--:|:--:|
| 忠实_直接引用事实 | 忠实 | 忠实 | ✅ |
| 忠实_合理转述 | 忠实 | 忠实 | ✅ |
| 幻觉_编造不存在的功能 | 幻觉 | 幻觉 | ✅ |
| 幻觉_数字篡改 | 幻觉 | 幻觉 | ✅ |

**裁判检出的具体幻觉（精准定位到声明级别）：**
- 编造功能案例 → `支持多人实时协作`、`还有安卓 App`
- 数字篡改案例 → `付费版每年 999 元`、`免费版可同步 30 台设备`

**结论**：该裁判在本样本集上达到门禁标准，可作为幻觉检测的一道关卡。

---

## 三、排错纪实（一次真实的"守门人"实战）

第一次运行**没有**顺利通过，而是一路报错——这段过程恰恰证明了"质量守门"的价值：

| 阶段 | 现象 | 真相 |
|---|---|---|
| 1 | `'str' object has no attribute 'choices'` | 响应不是标准对象，而是一段字符串 |
| 2 | JSON 解析失败 | 返回的根本不是模型输出 |
| 3 | 4 条全判 False，准确率 50% | 解析器误读了某段 JSON |
| 4 | 打印原始输出 | **竟是一个第三方网关的 HTML 首页** |
| 5 | 检查环境变量 | **shell 里残留的 `OPENAI_BASE_URL` 覆盖了 `.env`**，请求打到了网关首页而非 `/v1` 接口 |

**根因**：`load_dotenv()` 默认不覆盖已存在的 shell 环境变量，导致一个中转网关地址悄悄生效——**模型一次都没真正运行**。

**修复**：
1. `load_dotenv(override=True)` 让 `.env` 优先；清理残留 shell 变量。
2. 给测试加"端点健康检查"：返回 HTML 即当场报错，而非静默判错。
3. 强制要求响应含 `faithful` 字段，杜绝"缺字段默默当 False"。

> 💡 **守门人洞察**：我以为在测 deepseek，其实模型一次都没跑——是测试的护栏替我抓出了**配置错误**。一个连"自己有没有真正调用模型"都验证不了的评估流程，是不可信的。**评估系统本身，也需要被守门。**
